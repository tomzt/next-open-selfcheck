> ไม่ใช่การตัดสินใจด้านการออกแบบที่ยืนยันแล้ว — เป็นบันทึกความเป็นไปได้จากการพูดคุย
> สำรวจแนวทาง เก็บไว้เพื่อไม่ต้องคิดใหม่ทั้งหมดในภายหลัง อัปเดตหรือย้ายเนื้อหาไปไว้ใน

>
> **Source of truth คือ [`future-considerations.md`](./future-considerations.md)
> (ภาษาอังกฤษ)** — ไฟล์นี้เป็นฉบับแปล หากเนื้อหาไม่ตรงกันให้ยึดฉบับอังกฤษเป็นหลัก

## 1. ปัญหาที่ต้องการแก้

รูปแบบฮาร์ดแวร์ kiosk ปัจจุบัน (`docs/architecture.md`) คือ Mini PC + จอทัชสกรีน
1 ชุดต่อ 1 ไซต์ รันชุด Docker เต็มรูปแบบในเครื่องนั้นเลย ซึ่งอาจเป็นฮาร์ดแวร์ที่มากเกิน
ความจำเป็นสำหรับหลายกรณีการ deploy แนวคิดด้านล่างสำรวจว่าจะลดต้นทุนตรงนี้ได้ไหม

## 2. ทางเลือก: Android Tablet เป็น Thin Client

แทนที่ Mini PC + จอทัชแยกต่างหาก ด้วย Android tablet เครื่องเดียวที่รัน browser
ในโหมด kiosk

**ทำได้ทันที ไม่ต้องแก้โค้ดเลย สำหรับ Phase 1-3b (บาร์โค้ดอย่างเดียว):**
- UI พร้อมสำหรับ tablet อยู่แล้ว — `apps/kiosk/src/app/globals.css` ใช้
  `100vw/100vh` + flex layout ทั้งหมด ไม่มี fixed pixel width แบบเดสก์ท็อป
  ปุ่มกดมีขนาด ≥48px อยู่แล้ว
- `NEXT_PUBLIC_INPUT_MODE=touch` (`src/lib/theme.ts`, `globals.css`) ซ่อน
  cursor และปิด `:hover` อยู่แล้ว — สร้างไว้ด้วยเหตุผลอื่นแต่พอดีตรงกับที่
  tablet ต้องการ
- การสแกนบาร์โค้ดเป็นแบบ HID keyboard emulation ล้วนๆ (`onKeyDown`/Enter ใน
  `AuthScreen.tsx`, `BorrowScreen.tsx`, `ReturnScreen.tsx`) — เครื่องสแกน
  USB-OTG หรือ Bluetooth HID ใช้ได้กับโค้ดนี้โดยไม่ต้องแก้อะไร
- ไม่ต้องมี PWA/manifest — แอป kiosk-lockdown browser (เช่น "Fully Kiosk
  Browser") ชี้ไปที่ URL เดิมก็ใช้ได้เลย

**ราคา:** ~$100–200 (Android tablet ระดับ kiosk-grade แบบ budget-mid)
เทียบกับ ~$450–1,000 (NUC + จอทัช kiosk-grade) — ถูกกว่าประมาณ 2-4 เท่า
ที่ระดับใกล้เคียงกัน ถ้าเป็น Android tablet ระดับ commercial-grade (ทนใช้งาน
24/7) ราคาจะอยู่ที่ $300–500+ ทำให้ส่วนต่างแคบลง

**ข้อจำกัดสำคัญสำหรับ Phase 4 RFID:** ส่วน shared packages ใน
`docs/architecture.md` ระบุว่า `rfid-adapter/web-serial` ถูกใช้โดย
**ทั้ง Kiosk และ Workstation** — RFID มีแผนใช้งานบนตัว kiosk เอง ไม่ใช่แค่
เครื่องมือของเจ้าหน้าที่ **Web Serial API ไม่มีใน browser มือถือเลยไม่ว่ายี่ห้อไหน**
(รวม Chrome บน Android) เป็นข้อจำกัดระดับแพลตฟอร์ม ตอนนี้ยังไม่กระทบอะไร
(RFID ยังไม่ถูกสร้าง) แต่จำกัดให้ tablet fleet ใช้ได้แค่บาร์โค้ดเท่านั้น
เว้นแต่จะสร้าง native app ทีหลัง (ดู §5)

**ทำได้ในเว็บแอปเดิมด้วย (ไม่ต้องรื้อเป็น native):** อ่าน QR/บาร์โค้ดผ่าน
กล้องหน้าด้วย `getUserMedia` + library ถอดรหัส (เช่น `BarcodeDetector`
ในตัว browser หรือ `zxing-js`) แนะนำให้ใช้แค่กับ **การ login ด้วย QR
เท่านั้น** — กล้องหน้าถูกออกแบบมาสำหรับ video call (มุมกว้าง โฟกัสคงที่)
ไม่ใช่การถอดรหัสระยะใกล้ ดังนั้นการสแกนบาร์โค้ดหนังสือซ้ำๆ ในโหมด batch
scan น่าจะช้ากว่าและพลาดง่ายกว่าเครื่องสแกนจริง แนะนำให้เก็บเครื่องสแกน
HID ภายนอก (~$20–80) ไว้สำหรับส่วนยืม-คืน

## 3. ทางเลือก: iPad แทน Android

**ราคา:** สูงกว่า Android — iPad รุ่นถูกสุดที่ยังใหม่ ~$329–449 เทียบกับ
Android tablet ที่ $100–200 — กินเข้าไปในเป้าหมายประหยัดต้นทุนเดิม
ข้อแลกเปลี่ยน: ระยะเวลาซัพพอร์ต OS ของ Apple มักอยู่ที่ 5-7 ปี เทียบกับ
Android tablet ทั่วไปที่ 2-3 ปี ซึ่งอาจคุ้มกว่าราคาตั้งต้นที่สูงกว่าถ้ามองเป็น
การใช้งานระยะยาวหลายปี

**เรื่อง browser เหมือนกับ Android ทุกประการสำหรับความต้องการของแอปนี้:**
Apple บังคับให้ browser ทุกตัวบน iOS ต้องใช้ WebKit engine (แม้แต่ "Chrome
for iOS" ก็เป็น WebKit ข้างในเหมือนกัน) — Safari/WebKit ไม่รองรับ Web
Serial เช่นกัน ดังนั้น RFID ผ่าน browser ถูกบล็อกเท่ากันทั้งสองแพลตฟอร์ม
ไม่มีใครได้เปรียบ `getUserMedia` (สแกนผ่านกล้อง) และเครื่องสแกน HID
ภายนอก (ผ่าน adapter USB-C/Lightning หรือ Bluetooth) ใช้ได้ปกติบน iPad
เป็น pattern ที่ใช้กันมานานแล้วในวงการ POS/ร้านค้า

**จุดที่ iPad ต่างออกไป — เส้นทาง native RFID ถูกจำกัดด้วย MFi:** native
app ที่จะคุยกับอุปกรณ์ USB/Lightning ต้องผ่านการรับรอง MFi (Made for
iPhone/iPad) ของ Apple ผ่าน External Accessory framework — เป็น
กระบวนการปิดที่ผู้ผลิตต้องสมัครเข้าร่วมเอง ต่างจาก USB Host API ของ
Android ที่เปิดกว่า ตรวจสอบโดยตรงแล้ว: หน้าเว็บสินค้าของ ACS สำหรับ
**ACR1552U** (เครื่องอ่าน RFID ที่โปรเจกต์นี้อ้างอิงไว้) ระบุว่ารองรับ
**iOS/iPadOS 16 ขึ้นไป** — เป็นสัญญาณที่ดี แต่ไม่มีแหล่งข้อมูลสาธารณะ
ยืนยันว่ามี SDK สำหรับนักพัฒนาภายนอกจริง (ต่างจากแค่ "อุปกรณ์ใช้งานกับ
iOS ได้" ในระดับผู้บริโภคทั่วไป) **ต้องขอคำยืนยันโดยตรงจาก ACS ก่อน**
ถ้าจะพึ่งพา iPad สำหรับเส้นทาง native RFID

**การล็อกหน้าจอ kiosk:** "Guided Access" ที่มีในตัว iOS เทียบเท่ากับ
ตัวเลือก kiosk/screen-pinning ของ Android — ไม่มีความต่างที่มีนัยสำคัญ

## 4. ทางเลือก: รวมเซิร์ฟเวอร์ backend เป็นศูนย์กลาง (แยกอิสระจากเรื่อง tablet vs Mini PC)

เคยคิดไว้ตั้งแต่แรกแล้วพักไว้ — ควรกลับมาพิจารณาใหม่โดยแยกประเด็นให้ชัด
ตามด้านล่าง

**ข้อจำกัดจริงจาก RFID แคบกว่าที่คิดว่า "ห้ามมี server ร่วมกัน":** Web
Serial ต้องการแค่ให้ *browser* รันอยู่เครื่องเดียวกับ USB reader — ไม่ได้
ต้องการให้ทั้ง Docker/Postgres/SIP2 stack รันในเครื่องนั้นด้วย
`docs/architecture.md:21` / `docs/requirements.md:322` ที่เขียนว่า
"no central server, no WAN dependency" เป็น **การตัดสินใจด้าน resilience
ที่แยกต่างหากและตั้งใจเลือกเอง** (ไซต์หนึ่งยังทำงานต่อได้แม้ server กลาง
หรือ WAN link ล่ม) — ไม่มีที่ไหนเขียนไว้ว่าเป็นผลมาจาก RFID

**มีโครงสร้างที่ 3 ที่เป็นไปได้:** เซิร์ฟเวอร์ backend
(Docker/Postgres/SIP2) ตัวเดียวใช้ร่วมกัน + กล่องราคาถูกที่แต่ละ kiosk
รันแค่ browser (สำหรับให้ RFID เข้าถึง USB ในเครื่อง) ชี้ไปที่ backend
ส่วนกลาง วิธีนี้ยังให้ RFID ทำงานได้และลดการซ้ำซ้อนของฮาร์ดแวร์ backend
แต่ต้องยอมแลก "แต่ละไซต์อยู่รอดได้อิสระ" ทิ้งไป — เป็นข้อแลกเปลี่ยนจริงที่
ต้องตัดสินใจอย่างตั้งใจ ไม่ใช่ได้มาฟรีๆ

**Tablet ไม่เปลี่ยนสมการนี้สำหรับเรื่อง RFID โดยเฉพาะ** — ไม่ว่าจะรวม
ศูนย์หรือไม่ Web Serial ก็ยังใช้ไม่ได้บน mobile browser อยู่ดี ดังนั้น
tablet + RFID ยังต้องพึ่งเส้นทาง native app จาก §2/§3 เสมอไม่ว่าโครงสร้าง
backend จะเป็นแบบไหน

## 5. ทางเลือก: native app (Flutter) แทนเว็บแอป

เกี่ยวข้องเฉพาะถ้าต้องการ RFID บน tablet จริงๆ

- ปลดล็อกการเข้าถึงฮาร์ดแวร์โดยตรง: USB Host API ของ Android (ไม่มีข้อ
  จำกัดแบบ Web Serial) ส่วน iPad ถูกจำกัดด้วย MFi (ดู §3)
- ชั้น backend/API (`apps/kiosk/src/app/api/*`, SIP2 client, auth, email
  receipt, rate limiting) **ไม่ต้องเปลี่ยนเลย** — Flutter app จะเป็นแค่
  client ตัวใหม่ที่เรียก API endpoint เดิมที่มีอยู่แล้ว
- ส่วนที่เหลือทั้งหมดต้องสร้างใหม่: ทุกหน้า/component, ระบบ 5 theme,
  i18n, accessibility bar, session-timeout UI — เป็นงานขนาดใกล้เคียงกับ
  ตอนสร้างเว็บแอปครั้งแรก ไม่ใช่แค่แก้เล็กน้อย ระบบ auth ก็ต้องเปลี่ยนจาก
  cookie-based session เป็น token-based (เก็บ JWT ใน secure storage +
  ส่งผ่าน Authorization header)

## 6. ลำดับงานที่แนะนำ

| งาน | เวลาที่ควรทำ | เหตุผล |
|---|---|---|
| Pilot: ทดสอบ Android tablet 1 เครื่องเป็น thin client (บาร์โค้ดอย่างเดียว, ใช้ backend เดิม) | คู่ขนานได้ทุกเมื่อ | เป็นงานฮาร์ดแวร์/ปฏิบัติการล้วนๆ ไม่ใช้เวลา dev เลย ลดความเสี่ยงเรื่อง Wi-Fi/เครื่องสแกน/ขาตั้ง ก่อนซื้อจำนวนมาก |
| สแกน QR ด้วยกล้องหน้าสำหรับ login | คู่ขนานได้ เมื่อสะดวก | เป็นฟีเจอร์เว็บเล็กๆ แยกออกจากส่วนที่เกี่ยวกับ RFID |
| ตัดสินใจ iPad vs Android ต่อไซต์ | คู่ขนานไปกับ pilot | เป็นเรื่อง trade-off งบประมาณ vs อายุการใช้งาน เช็คกับ ACS เรื่อง iOS SDK ของ ACR1552U ถ้าไซต์นั้นต้องการ RFID |
| รื้อเป็น native (Flutter) + RFID บน tablet | **หลังจาก** Phase 4 RFID สร้างเสร็จและพิสูจน์แล้วบน Mini PC | หลีกเลี่ยงการ debug ฮาร์ดแวร์ใหม่ (protocol RFID/การเขียน AFI) พร้อมกับ framework ใหม่ในเวลาเดียวกัน |

## 7. คำถามที่ต้องตอบให้ได้ก่อนใช้งบประมาณจริง

- [ ] ยืนยันกับ ACS ว่า ACR1552U มี iOS SDK สาธารณะ/ให้สิทธิ์ใช้งานสำหรับ third-party app จริงไหม หรือมีแค่ MFi ที่จำกัดเฉพาะ first-party
- [ ] สำรวจหน้างาน: Wi-Fi ที่แต่ละจุดติดตั้ง kiosk เสถียรพอไหม หรือต้องเดินสายไปยัง access point ใกล้ๆ ขาตั้ง
- [ ] ตัดสินใจต่อไซต์: แยก Docker stack อิสระ (แบบปัจจุบัน ทนทานกว่า) vs รวม backend ส่วนกลาง (ถูกกว่า แต่มีจุดล้มเหลวจุดเดียว) — ดู §4
- [ ] ยืนยันว่า tablet รุ่นที่เลือกรองรับ USB-OTG จริง (ไม่ใช่ Android tablet ราคาถูกทุกรุ่นจะรองรับ) ถ้าจะใช้เครื่องสแกนแบบมีสายแทน Bluetooth
- [ ] เผื่องบสำหรับขาตั้ง/กรอบล็อกกันขโมย ไม่ว่าจะเลือกแพลตฟอร์มไหน
